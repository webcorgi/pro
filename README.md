# Spec Driven Design (SDD) with LLM

대규모 언어 모델을 활용하여 Spec Driven Design 방법론을 구현하기 위한 구조화된 프롬프트 컬렉션입니다. 이 저장소는 아이디어를 명확하게 정의된 요구사항, 포괄적인 설계, 실행 가능한 구현 계획으로 변환하기 위한 체계적인 접근 방식을 제공합니다.

## 개요

Spec Driven Design은 기능 개발을 세 가지 고유한 단계로 나누는 방법론입니다:

1. **요구사항 수집** - 아이디어를 상세하고 테스트 가능한 요구사항으로 변환
2. **설계 작성** - 요구사항을 기반으로 포괄적인 기술 설계 개발
3. **작업 계획** - 개발자를 위한 실행 가능한 구현 계획 작성

이 접근 방식은 코딩을 시작하기 전에 철저한 계획을 보장하여 개발 시간을 단축하고 코드 품질을 향상시킵니다.

## 저장소 구조

```text
kiro-style-sdd/
├── README.md          # 이 파일
├── spec.md           # 1단계: 요구사항 수집 프롬프트
├── design.md         # 2단계: 설계 작성 프롬프트  
├── tasks.md          # 3단계: 작업 계획 프롬프트
├── implement.md      # 구현 가이드 프롬프트
└── specs/            # 프로젝트 사양서 (필요시 생성)
    └── your_project/ # 예제 프로젝트 폴더 구조
        ├── requirements.md
        ├── design.md
        └── tasks.md
```

## 시작하기

### 1단계: 프로젝트 폴더 만들기

`specs/` 디렉토리 아래에 프로젝트를 위한 새 폴더를 만듭니다:

```bash
mkdir -p specs/your_project_name
cd specs/your_project_name
```

`your_project_name`을 기능이나 애플리케이션에 맞는 설명적인 이름으로 바꾸세요.

### 2단계: 요구사항 단계

1. 선호하는 LLM과 **새 대화를 시작**하세요 (**Gemini 2.5 Pro**와 특히 잘 작동합니다)
2. `spec.md`의 **전체 내용을 복사하여 붙여넣기**하세요
3. 요청받으면 **기능 아이디어를 설명**하세요
4. **생성된 requirements.md를 신중하게 검토**하세요
5. 완전히 만족할 때까지 **명확화나 수정을 요청**하세요
6. **최종 requirements.md를 프로젝트 폴더에 저장**하세요

**예시 대화:**

```text
사용자: [spec.md 내용 붙여넣기]
사용자: 실시간 협업 기능이 있는 작업 관리 앱을 만들고 싶습니다

LLM: [초기 requirements.md 생성]
사용자: 사용자 인증과 파일 첨부에 대한 요구사항을 추가해주실 수 있나요?
LLM: [requirements.md 업데이트]
사용자: 요구사항이 잘 보입니다! 설계로 넘어가겠습니다.
```

### 3단계: 설계 단계

1. **동일한 대화에서** (컨텍스트 유지를 위해) `design.md`의 내용을 복사하여 붙여넣기하세요
2. **생성된 design.md를 철저히 검토**하세요
3. 필요에 따라 **수정을 요청**하세요
4. 만족할 때까지 **설계를 승인하지 마세요**
5. **최종 design.md를 프로젝트 폴더에 저장**하세요

**중요:** 요구사항과의 컨텍스트를 유지하기 위해 동일한 채팅 대화를 사용하세요.

### 4단계: 작업 계획 단계

1. **동일한 대화에서 계속 진행**하고, `tasks.md`의 내용을 복사하여 붙여넣기하세요
2. **생성된 작업 목록을 검토**하세요
3. 실행 가능하고 완전할 때까지 **작업을 개선**하세요
4. **최종 작업 계획을 승인**하세요
5. **최종 tasks.md를 프로젝트 폴더에 저장**하세요

### 5단계: 구현

세 개의 문서(requirements.md, design.md, tasks.md)를 모두 가지고 있으면:

1. **어떤 코딩 어시스턴트라도 사용**하세요 (Claude, ChatGPT, Copilot 등)
2. `implement.md`의 **내용을 복사하여 붙여넣기**하세요
3. 세 개의 사양 문서를 포함하는 **프로젝트 폴더 경로를 제공**하세요
4. **tasks.md 파일에서 특정 작업을 구현하도록 요청**하세요

**예시:**

```text
사용자: [implement.md 내용 붙여넣기]
사용자: /path/to/specs/my_project/의 tasks에서 작업 2.1을 구현해주세요
```

## 모범 사례

### ✅ 해야 할 것

- **컨텍스트 유지** - 동일한 대화를 사용하여 요구사항 → 설계 → 작업 순서로 진행
- **철저히 검토** - 각 단계를 다음 단계로 넘어가기 전에 꼼꼼히 검토
- **명확하게 요청** - 변경이나 명확화를 요청할 때 구체적으로 요청
- **문서 저장** - 각 단계 완료 후 문서 저장
- **한 번에 하나씩 구현** - 코딩 단계에서 작업을 하나씩 구현

### ❌ 하지 말아야 할 것

- **단계 건너뛰지 않기** - 각 단계는 이전 단계를 기반으로 합니다
- **승인에 성급하지 않기** - 철저한 검토가 나중에 시간을 절약합니다
- **단계 사이에 새 대화 시작하지 않기** (컨텍스트 손실)
- **여러 작업을 동시에 구현하지 않기**
- **문서를 수동으로 수정하지 않기** - 일관성을 위해 LLM 사용

## 예시 워크플로우

다음은 SDD 프로세스의 완전한 예시입니다:

### 1. 요구사항 단계

```markdown
📁 specs/todo_app/
└── requirements.md ✅ (생성 및 승인됨)
```

### 2. 설계 단계  

```markdown
📁 specs/todo_app/
├── requirements.md ✅
└── design.md ✅ (생성 및 승인됨)
```

### 3. 작업 계획 단계

```markdown
📁 specs/todo_app/
├── requirements.md ✅
├── design.md ✅
└── tasks.md ✅ (생성 및 승인됨)
```

### 4. 구현 단계

```markdown
📁 specs/todo_app/
├── requirements.md ✅
├── design.md ✅
├── tasks.md ✅
└── src/ (구현 시작)
    ├── models/
    ├── services/
    └── tests/
```

## 이 방법이 효과적인 이유

### 체계적인 접근 방식

- **구조화된 사고** - 중요한 측면을 놓치지 않도록 방지
- **반복적 개선** - 각 단계에서 품질 보장
- **컨텍스트 보존** - 단계 전반에 걸쳐 일관성 유지

### LLM 최적화

- **구체적인 프롬프트** - LLM이 일관되고 고품질의 출력을 생성하도록 안내
- **단계 분리** - 너무 많은 컨텍스트로 모델을 압도하는 것을 방지
- **승인 게이트** - 중요한 결정 지점에서 인간의 감독 보장

### 개발 이점

- **재작업 감소** - 철저한 사전 계획을 통해
- **명확한 구현 경로** - 실행 가능한 작업과 함께
- **더 나은 추정** - 상세한 작업 분해에서
- **코드 품질 향상** - 테스트 주도 접근을 통해

## 추천 LLM

이 방법론은 대부분의 최신 LLM에서 작동하지만 다음을 추천합니다:

1. **Gemini 2.5 Pro** - 전체 SDD 워크플로우에 우수
2. **Claude 3.5 Sonnet** - 요구사항 및 설계 단계에 적합
3. **ChatGPT-4** - 전반적으로 좋은 성능
4. **코딩 중심 LLM** - 구현 단계용

## 문제 해결

### 일반적인 문제

**Q: LLM이 프롬프트 구조를 따르지 않습니다**

- 프롬프트 내용 전체를 복사했는지 확인하세요
- 기능 설명을 다른 방식으로 표현해보세요
- 다른 LLM 모델을 사용하세요

**Q: 요구사항이 불완전해 보입니다**

- 경계 사례에 대해 구체적인 질문을 하세요
- 추가 사용자 스토리를 요청하세요
- 기술적 제약 사항에 대한 명확화를 요청하세요

**Q: 설계가 요구사항과 일치하지 않습니다**

- 구체적인 불일치를 지적하세요
- LLM에게 요구사항을 다시 검토하도록 요청하세요
- 특정 섹션에 대한 설계 수정을 요청하세요

**Q: 작업이 너무 모호하거나 너무 세부적입니다**

- 더 구체적인 구현 단계를 요청하세요
- 작업 분해나 통합을 요청하세요
- 개별 작업의 범위를 명확히 하세요
