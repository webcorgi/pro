## 📋 포맷 규칙 (Formatting Rules)

- 작업 목록은 **체크박스 형식의 번호 리스트**로 구성합니다.  
- 계층은 최대 **2단계**까지만 허용합니다.  
  - 1단계: 큰 기능 단위(Epic)  
  - 2단계: 세부 하위 작업(Sub-task, 예: `1.1`, `1.2`)  
- 각 작업은 **명확한 코드 관련 목표(Objective)** 를 포함해야 합니다.  
- 각 항목에는 다음을 포함합니다:  
  - **작업 설명:** 작성/수정/테스트해야 할 코드 목적  
  - **보조 정보(Sub-bullets):** 관련 세부 내용  
  - **참조 요구사항:** 요구사항 문서 내 세부 항목 번호를 `_Requirements: 1.2_` 형식으로 표시  

---

## 🧩 세부 작성 지침 (Detailed Task Creation Guidelines)

- 모든 작업은 **관리 가능한 코드 단위**여야 합니다.  
- 각 단계는 이전 단계의 결과를 **점진적으로 확장**해야 합니다.  
- 설계 문서에 이미 포함된 구현 세부사항을 **중복 기재하지 않습니다.**  
- 모든 맥락 문서(요구사항, 설계)는 구현 중 참조 가능한 상태라고 가정합니다.  
- 테스트 주도 개발(TDD) 전략을 우선 고려합니다.  
- 초기 단계에서 핵심 기능(Core Functionality)을 조기에 검증하도록 순서를 구성합니다.  
- 모든 요구사항이 구현 계획에 포함되어야 합니다.  
- 계획 수립 중 누락된 부분이 발견되면,  
  → **요구사항 또는 설계 단계로 되돌아가야 합니다.**

## 🚫 포함해서는 안 되는 비코딩 작업 (Non-coding Tasks to Avoid)

다음 유형의 작업은 **절대 포함하지 않습니다:**

- 사용자 테스트(UAT) 또는 사용자 피드백 수집  
- 스테이징/프로덕션 배포  
- 성능 측정 또는 로깅/분석  
- 애플리케이션을 수동으로 실행하여 전체 플로우를 테스트하는 작업  
  (단, 자동화된 테스트 코드는 작성 가능)  
- 사용자 교육 또는 문서화 작업  
- 비즈니스 프로세스/조직 변화  
- 마케팅 또는 커뮤니케이션 관련 활동  
- 코드 작성·수정·테스트 외의 모든 활동

## ✅ 작업 실행 지침 (Actionability Requirements)

모든 작업은 **코딩 에이전트가 즉시 실행 가능한 수준**으로 구체적이어야 합니다.

- 반드시 코드 작성, 수정 또는 테스트가 포함되어야 함  
- 생성 또는 수정해야 할 파일/컴포넌트를 명시  
- 추가 설명 없이 실행 가능할 정도로 구체적이어야 함  
- 개념적 수준이 아닌, **구현 중심(Implementation-focused)** 이어야 함  
- 가능한 한 세분화된 단위로 구성 (예: “X 기능 구현”이 아닌 “X 함수 구현”)  

---

## 🧾 승인 절차 (Approval Process)

- 작업 문서(tasks.md)를 업데이트한 후에는 반드시 다음과 같이 물어야 합니다:  
  > “작업 목록이 괜찮아 보이나요?”  

- 사용자가 변경을 요청하거나 명시적으로 승인하지 않은 경우,  
  → **문서를 수정해야 합니다.**  

- 모든 수정 후에는 **명시적 승인(explicit approval)** 을 받아야 합니다.  
- 승인받기 전까지는 **다음 단계로 진행할 수 없습니다.**

## 📘 예시 포맷 (Example Format - 축약본)

```markdown
# Implementation Plan

- [ ] 1. 프로젝트 구조 및 핵심 인터페이스 설정  
  - 모델, 서비스, 리포지토리, API 컴포넌트 디렉터리 구조 생성  
  - 시스템 경계를 정의하는 인터페이스 설계  
  - _Requirements: 1.1_

- [ ] 2. 데이터 모델 및 유효성 검사 구현  
- [ ] 2.1 핵심 데이터 모델 인터페이스 및 타입 생성  
  - 모든 데이터 모델의 TypeScript 인터페이스 작성  
  - 데이터 무결성 검증 함수 구현  
  - _Requirements: 2.1, 3.3, 1.2_

- [ ] 2.2 사용자(User) 모델 구현  
  - User 클래스 및 검증 메서드 작성  
  - User 모델 검증용 단위 테스트 추가  
  - _Requirements: 1.2_

- [ ] 2.3 문서(Document) 모델 및 관계 설정  
  - 관계형 Document 클래스 작성  
  - 관계 관리용 단위 테스트 작성  
  - _Requirements: 2.1, 3.3, 1.2_

- [ ] 3. 저장소 메커니즘 구현  
- [ ] 3.1 데이터베이스 연결 유틸리티 작성  
  - 연결 관리 코드 작성  
  - DB 연산용 오류 처리 유틸리티 구현  
  - _Requirements: 2.1, 3.3, 1.2_

- [ ] 3.2 리포지토리 패턴 구현 (데이터 접근 계층)  
  - 기본 리포지토리 인터페이스 정의  
  - CRUD 연산이 가능한 구체 리포지토리 작성  
  - 리포지토리 단위 테스트 작성  
  - _Requirements: 4.3_

[이후 추가 구현 작업이 계속됩니다...]
